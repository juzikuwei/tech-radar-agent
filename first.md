# AI/Agent 技术雷达助手（Tech Radar Agent）

## 1. 项目定位

这是一个用于学习和实践 RAG、Agent、MCP、评测与可观测性的渐进式项目。

项目最终希望成为一个面向 AI/Agent 技术领域的知识助手，可以收集公开技术资料，并回答下面这类问题：

- 某项 AI/Agent 技术是什么？
- 某个方向最近有哪些论文？
- 两种技术有什么区别？
- 当前本地知识库是否包含足够资料？

项目不会一次性实现所有功能，而是先完成一个可以运行、可以理解、可以测试的基础 RAG，再逐步增加 Agent 能力。

---

## 2. 开发与学习原则

本项目采用“教学式共建”的方式推进：

1. 每次只完成一个较小、可运行的目标。
2. 编写代码前先解释当前模块解决什么问题。
3. 用户参与关键代码、命令执行、测试和技术选择。
4. 每完成一步，都验证输入、输出和失败情况。
5. 基础流程没有跑通之前，不提前引入复杂框架。
6. 新增功能时说明它解决了原有系统的什么不足。
7. 除非明确要求，不一次性生成整个项目。

学习重点不是简单调用框架，而是理解以下问题：

- 外部数据如何进入系统？
- 文档为什么需要向量化？
- 语义检索和关键词搜索有什么区别？
- RAG 为什么仍然可能产生幻觉？
- 如何判断资料不足并拒绝回答？
- 普通 RAG 在什么情况下需要升级成 Agentic RAG？
- LangGraph、MCP、Eval 和 Observability 分别解决什么问题？

## 4. 第一版目标：arXiv 本地基础 RAG

第一版只使用一个数据源：**arXiv**。

第一版不是完整 Agent，而是一个基础的本地 RAG 系统。先把数据抓取、数据存储、向量检索、回答生成和引用这条链路做正确，再逐步加入 Agent 能力。

### 第一版需要实现

- 手动批量抓取 arXiv 论文。
- 将抓取结果一次性保存到本地。
- 按 arXiv ID 去重，允许以后重新运行抓取程序。
- 将论文标题和摘要转换为向量并保存。
- 支持使用中文或英文问题检索英文论文。
- 根据本地检索结果生成中文回答。
- 回答中展示论文来源和 arXiv 链接。
- 本地资料不足时明确拒答。
- 使用 Streamlit 提供基础网页界面。

### 第一版暂时不做

- 不在用户每次提问时实时请求 arXiv。
- 不下载和解析论文 PDF。
- 不做定时任务。
- 不接入 Hugging Face Hub。
- 不使用 LangGraph。
- 不实现 MCP Server。
- 不拆分多个 Agent。
- 不加入复杂的反思循环。
- 不接入 Langfuse 或 LangSmith。
- 不做复杂的云端部署。

这些功能会在基础版本稳定后逐步增加。

## 5. arXiv 数据方案

### 5.1 获取模式

使用 arXiv 官方 API，并采用“手动批量抓取、保存本地、问答阶段不访问 arXiv”的模式。

```text
手动运行抓取程序
        ↓
arXiv 官方 API
        ↓
保存 JSONL 抓取快照
        ↓
清洗并按 arXiv ID 去重
        ↓
写入 SQLite
        ↓
生成向量并写入 ChromaDB
        ↓
用户问答只访问本地数据
```

这种模式可以降低系统复杂度，也可以避免用户问答时高频访问 arXiv 而触发限流。

### 5.2 初始关注范围

主要关注分类：

- `cs.AI`：人工智能
- `cs.CL`：计算语言学与大语言模型
- `cs.MA`：多智能体系统

初始关键词通过配置文件管理，例如：

- agent
- LLM agent
- retrieval augmented generation
- RAG
- tool use
- multi-agent
- reasoning agent

第一阶段会从很小的抓取数量开始，例如先获取 3 篇论文验证 API，再逐步增加到每个查询 50～100 篇。

### 5.3 限流与可靠性

- API 请求之间保留合理间隔。
- 每次抓取设置最大结果数量。
- 请求失败时进行有限次数重试。
- 保存 JSONL 抓取快照，同时保留源文本和规范化文本。
- 使用 arXiv ID 作为论文唯一标识。
- 重复运行时使用 upsert，而不是重复插入。

### 5.4 保存字段

第一版保存以下字段：

- `arxiv_id`
- `title`
- `abstract`
- `authors`
- `categories`
- `primary_category`
- `published_at`
- `updated_at`
- `entry_url`
- `pdf_url`
- `query_keyword`
- `fetched_at`

第一版只使用标题和摘要建立知识库，不解析 PDF 正文。

## 6. 第一版技术选型

| 模块 | 技术选择 | 选择原因 |
|---|---|---|
| Python | Python 3.12 | 与本地环境一致 |
| arXiv 抓取 | `arxiv` Python 包 | 基于官方 API，调用简单 |
| 抓取快照 | JSONL | 保留源文本与规范化文本，便于追溯和重新处理 |
| 结构化数据 | SQLite | 本地零配置，方便去重和查询 |
| 向量数据库 | ChromaDB | 本地持久化，适合学习和起步 |
| 嵌入模型 | `intfloat/multilingual-e5-small` | 支持中文问题与英文摘要的跨语言检索 |
| 大模型 | OpenAI-compatible 接口 | 通过环境变量切换模型供应商 |
| 前端 | Streamlit | 可以快速展示检索、回答和引用 |
| 测试 | pytest | 验证抓取、去重和检索行为 |

第一阶段不会立刻安装表格中的全部依赖。依赖会随着学习阶段逐步加入，以便理解每个库的作用。

## 7. 基础 RAG 问答流程

```text
用户输入问题
      ↓
使用多语言嵌入模型生成问题向量
      ↓
从 ChromaDB 检索最相关的论文摘要
      ↓
判断检索结果是否达到最低相关度要求
      ├── 不足：明确说明本地知识库没有足够资料
      └── 足够：把检索内容交给大模型
                     ↓
               生成中文回答
                     ↓
               展示对应论文引用
```

第一版只做一次检索，不做自动改写查询、循环检索或工具选择。这些能力将在升级到 Agentic RAG 时加入。

- [x] 把抓取逻辑封装为独立模块。
- [x] 保存原子发布的 JSONL 抓取快照。
- [x] 使用稳定 arXiv ID 和规范化内容哈希。

验收标准：重复运行抓取程序不会产生重复论文。

### 阶段 2：本地语义检索

目标：理解 embedding、向量数据库和相似度检索。

- [x] 加载 `multilingual-e5-small` 多语言嵌入模型。
- [x] 为标题和摘要生成内存向量。
- [x] 建立本地持久化 ChromaDB 索引。
- [x] 按 `arxiv_id` 和 `content_hash` 增量同步向量。
- [x] 输入中文问题并返回最相关英文论文。
- [x] 展示检索分数和论文元数据。

验收标准：输入中文技术问题时，能够检索到相关英文摘要。

### 阶段 3：基础 RAG 回答

目标：理解检索增强生成和引用约束。

- 接入可配置的大模型接口。
- 只根据检索结果生成回答。
- 给回答添加论文引用。
- 校验引用是否来自实际检索结果。
- 检索结果不足时拒绝回答。

验收标准：回答中的引用真实存在，询问无关内容时系统不会强行编造。

### 阶段 4：Streamlit 页面

目标：把数据、检索和回答整合成可使用的产品。

- 提供问题输入框。
- 展示最终回答。
- 展示引用论文、链接和相似度。
- 展示知识库论文数量和更新时间。

验收标准：用户可以在网页中完成一次完整问答。

### 阶段 5：从普通 RAG 升级到 Agentic RAG

目标：理解 Agent 相比固定流程增加了什么能力。

依次加入：

1. [x] Query 改写。
2. [x] 检索质量判断。
3. [x] 第二次检索。
4. [x] 最大循环次数和终止条件。
5. [x] 记录每次请求的结构化执行轨迹。
6. [x] 保存最近六轮对话和上一轮活动证据，支持直接复用、补充检索和新话题检索。
7. LangGraph 状态图。
8. 回答前的引用检查和反思节点。

### 阶段 6：MCP 工具化

目标：理解模型、Agent 与外部工具之间的标准化调用方式。

计划实现：

- `query_knowledge_base(query, top_k)`
- `get_paper_by_arxiv_id(arxiv_id)`
- `get_knowledge_base_stats()`
- 手动触发的 arXiv 批量更新工具

实时问答仍优先使用本地知识库，避免把 arXiv API 变成高频实时搜索接口。

### 阶段 7：第二数据源 Hugging Face Hub

基础版本完成后，再加入 Hugging Face Hub：

- 使用 `huggingface_hub` Python 库。
- 通过 `HF_ENDPOINT=https://hf-mirror.com` 切换到镜像。
- 优先收集与 Agent、RAG、Embedding 相关的模型和 Space。
- 为不同数据源保留统一元数据格式。

### 阶段 8：评测与可观测性

- [x] 建立三组对话动作场景，自动比较证据复用、补充检索和新话题检索。
- 建立事实型、多跳型和拒答型测试问题。
- 统计引用命中率和拒答准确率。
- 接入 Langfuse 或 LangSmith。
- 记录检索、模型调用、耗时和 token 消耗。

## 10. 第一版完成标准

第一版完成时，应满足：

- arXiv 数据只通过手动批量任务获取。
- 论文元数据已经保存到本地。
- 重复抓取不会产生重复记录。
- 中文问题可以检索英文论文摘要。
- 回答只引用实际检索到的论文。
- 资料不足时会明确拒答。
- Streamlit 页面可以展示问答和引用。
- 至少有一组可以重复运行的测试问题。
