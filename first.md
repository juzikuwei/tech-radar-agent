# AI/Agent 技术雷达助手（Tech Radar Agent）

## 1. 项目定位

这是一个用于学习和实践 RAG、Agent、MCP 与可观测性的渐进式项目。

项目最终希望成为一个面向 AI/Agent 技术领域的知识助手，可以收集公开技术资料，并回答下面这类问题：

- 某项 AI/Agent 技术是什么？
- 某个方向最近有哪些论文？
- 两种技术有什么区别？
- 当前本地知识库是否包含足够资料？

项目不会一次性实现所有功能，而是先完成一个可以运行、可以理解、可以测试的基础 RAG，再逐步增加 Agent 能力。

---

## 2. 开发与学习原则

本项目采用“教学式共建”的方式推进：

1. 每次只完成一个较小、可运行的目标。
2. 编写代码前先解释当前模块解决什么问题。
3. 用户参与关键代码、命令执行、测试和技术选择。
4. 每完成一步，都验证输入、输出和失败情况。
5. 基础流程没有跑通之前，不提前引入复杂框架。
6. 新增功能时说明它解决了原有系统的什么不足。
7. 除非明确要求，不一次性生成整个项目。

学习重点不是简单调用框架，而是理解以下问题：

- 外部数据如何进入系统？
- 文档为什么需要向量化？
- 语义检索和关键词搜索有什么区别？
- RAG 为什么仍然可能产生幻觉？
- 如何判断资料不足并拒绝回答？
- 普通 RAG 在什么情况下需要升级成 Agentic RAG？
- LangGraph、MCP 和 Observability 分别解决什么问题？

## 4. 第一版目标：arXiv 本地基础 RAG

第一版只使用一个数据源：**arXiv**。

第一版不是完整 Agent，而是一个基础的本地 RAG 系统。先把数据抓取、数据存储、向量检索、回答生成和引用这条链路做正确，再逐步加入 Agent 能力。

### 第一版需要实现

- 手动批量抓取 arXiv 论文。
- 将抓取结果一次性保存到本地。
- 按 arXiv ID 去重，允许以后重新运行抓取程序。
- 将论文标题和摘要转换为向量并保存。
- 支持使用中文或英文问题检索英文论文。
- 根据本地检索结果生成中文回答。
- 回答中展示论文来源和 arXiv 链接。
- 本地资料不足时明确拒答。
- 使用 React、TypeScript 和 Vite 提供独立网页界面。

### 第一版暂时不做

- 不在用户每次提问时实时请求 arXiv。
- 不下载和解析论文 PDF。
- 不做定时任务。
- 不接入 Hugging Face Hub。
- 不使用 LangGraph。
- 不实现 MCP Server。
- 不拆分多个 Agent。
- 不加入复杂的反思循环。
- 不接入 Langfuse 或 LangSmith。
- 不做复杂的云端部署。

这些功能会在基础版本稳定后逐步增加。

## 5. arXiv 数据方案

### 5.1 获取模式

使用 arXiv 官方 API，并采用“手动批量抓取、保存本地、问答阶段不访问 arXiv”的模式。

```text
手动运行抓取程序
        ↓
arXiv 官方 API
        ↓
保存 JSONL 抓取快照
        ↓
清洗并按 arXiv ID 去重
        ↓
写入 SQLite
        ↓
生成向量并写入 ChromaDB
        ↓
用户问答只访问本地数据
```

这种模式可以降低系统复杂度，也可以避免用户问答时高频访问 arXiv 而触发限流。

### 5.2 初始关注范围

主要关注分类：

- `cs.AI`：人工智能
- `cs.CL`：计算语言学与大语言模型
- `cs.MA`：多智能体系统

初始关键词通过配置文件管理，例如：

- agent
- LLM agent
- retrieval augmented generation
- RAG
- tool use
- multi-agent
- reasoning agent

第一阶段会从很小的抓取数量开始，例如先获取 3 篇论文验证 API，再逐步增加到每个查询 50～100 篇。

### 5.3 限流与可靠性

- API 请求之间保留合理间隔。
- 每次抓取设置最大结果数量。
- 请求失败时进行有限次数重试。
- 保存 JSONL 抓取快照，同时保留源文本和规范化文本。
- 使用 arXiv ID 作为论文唯一标识。
- 重复运行时使用 upsert，而不是重复插入。

### 5.4 保存字段

第一版保存以下字段：

- `arxiv_id`
- `title`
- `abstract`
- `authors`
- `categories`
- `primary_category`
- `published_at`
- `updated_at`
- `entry_url`
- `pdf_url`
- `query_keyword`
- `fetched_at`

第一版只使用标题和摘要建立知识库，不解析 PDF 正文。

## 6. 第一版技术选型

| 模块 | 技术选择 | 选择原因 |
|---|---|---|
| Python | Python 3.12 | 与本地环境一致 |
| arXiv 抓取 | `arxiv` Python 包 | 基于官方 API，调用简单 |
| 抓取快照 | JSONL | 保留源文本与规范化文本，便于追溯和重新处理 |
| 结构化数据 | SQLite | 本地零配置，方便去重和查询 |
| 向量数据库 | ChromaDB | 本地持久化，适合学习和起步 |
| 嵌入模型 | `intfloat/multilingual-e5-small` | 支持中文问题与英文摘要的跨语言检索 |
| 大模型 | OpenAI-compatible 接口 | 通过环境变量切换模型供应商 |
| 后端 API | FastAPI | 为独立前端提供清晰、可测试的 HTTP 边界 |
| 前端 | React + TypeScript + Vite | 支持完整聊天交互、引用卡片和 Agent Trace |
| 测试 | pytest | 验证抓取、去重和检索行为 |

第一阶段不会立刻安装表格中的全部依赖。依赖会随着学习阶段逐步加入，以便理解每个库的作用。

## 7. 基础 RAG 问答流程

```text
用户输入问题
      ↓
使用多语言嵌入模型生成问题向量
      ↓
从 ChromaDB 检索最相关的论文摘要
      ↓
判断检索结果是否达到最低相关度要求
      ├── 不足：明确说明本地知识库没有足够资料
      └── 足够：把检索内容交给大模型
                     ↓
               生成中文回答
                     ↓
               展示对应论文引用
```

第一版只做一次检索，不做自动改写查询、循环检索或工具选择。这些能力将在升级到 Agentic RAG 时加入。

### 阶段 9：ReAct 循环 Agent

目标：理解"模型自主决定调用哪个工具、调用几次"与固定管线的本质区别。

背景：当前 `/chat` 是硬编码的固定管线（改写 → 检索 → 判断 → 二次检索 → 生成），检索策略由代码决定，不由模型决定。ReAct 让模型在"思考 → 选工具 → 看观察结果 → 再思考"的循环中自主行动，是当前生产环境最常见的 Agent 模式。

- [ ] 把现有三个 MCP 只读工具（`query_knowledge_base`、`get_paper_by_arxiv_id`、`get_knowledge_base_stats`）封装为进程内可调用的工具描述。
- [x] 实现一个不依赖框架的最小 ReAct 循环：把工具列表交给 LLM，解析工具调用，执行后把观察结果回填，直到模型给出最终回答。
- [x] 设置最大检索次数，规划或工具执行失败时降级到现有固定管线。
- [ ] 设置请求级 token 上限，超限时降级到现有固定管线。
- [x] 在执行轨迹中记录每一步的思考摘要、工具名、参数和观察结果，并在 React 前端展示。
- [x] 提供 `mode=pipeline | react` 开关，两种模式并存，不删除固定管线。

验收标准：可以用少量真实问题分别运行两种模式，并根据 Trace 解释 ReAct 为什么继续检索、停止、拒答或回退。

### 阶段 10：Reflection 引用审查与回答反思

目标：理解"生成后自我审查再修订"如何降低幻觉（对应阶段 5 未完成的第 8 项）。

**当前状态：暂缓。** 现有 ReAct 已经会在每次检索后观察证据、更新子问题状态，并决定继续检索、完成或拒答，已经覆盖当前小项目最有价值的过程反思。完整的 LLM 回答反思会为每次问答增加额外模型调用，目前没有足够的真实失败案例证明收益。

恢复条件：真实使用中多次出现错误引用、结论明显超出摘要范围，或者后续专门学习 Reflection 设计模式时，再实现以下内容：

- [ ] 生成回答后增加一个反思节点：用 LLM 审查回答中每条引用是否真实存在于本轮检索证据中、结论是否超出证据范围。
- [ ] 审查发现问题时执行一次修订：删除或替换无依据的句子，最多修订一次，避免无限循环。
- [ ] 修订失败或审查调用失败时降级为直接返回原回答，并在 Trace 中标注。
- [ ] 在执行轨迹中记录审查结论、发现的问题和是否修订。

验收标准：能通过一次真实 Trace 展示反思节点发现无依据内容并完成修订，失败时能安全降级。

### 阶段 11：LangGraph 状态图重构

目标：理解有状态、可中断、可回放的图式编排解决了手写编排的什么问题（对应阶段 5 未完成的第 7 项）。

**当前状态：暂缓。** 当前 ReAct 只有论文检索工具、有限循环和明确回退，手写编排仍然容易理解。等到出现更多工具、可中断任务、并行分支或人工审批后，再引入 LangGraph。

- [ ] 先不改代码，画出当前 `rag/pipeline.py` 编排的节点和边（改写、检索、判断、二次检索、重排、生成、反思）。
- [ ] 引入 LangGraph，把上述节点改写为显式 StateGraph，状态对象承载问题、证据、轨迹和循环计数。
- [ ] 保持对外接口不变：`/chat` 的请求和响应契约、执行轨迹格式不变，前端无需改动。
- [ ] 原手写编排保留为回退实现，通过配置切换，直到 LangGraph 版通过全部现有测试和真实问答验收。
- [ ] 在 `docs/decision-log.md` 记录引入框架的决策、代价和回退条件。

验收标准：LangGraph 版通过全部现有 pytest，并能从真实 Trace 验证主要分支行为与原实现一致。

### 阶段 12：技术雷达周报与多 Agent 验证

目标：先把现有本地知识库转化为有实际用途的技术雷达周报，再根据单 Agent 的真实限制判断是否需要 Supervisor-Worker 多 Agent。

- [ ] 先实现单 Agent 周报基线：给定时间窗口和主题列表，生成一份带 arXiv 引用的中文 Markdown 周报。
- [ ] 明确周报结构、论文筛选边界、资料不足处理和生成失败行为。
- [ ] 只有单 Agent 暴露出主题拆分、上下文长度或汇总质量问题后，再引入 Supervisor-Worker。
- [ ] 若引入多 Agent，由 Supervisor 拆分主题并汇总，每个 Worker 独立检索和总结。
- [ ] Worker 复用现有检索服务，不复制检索逻辑。
- [ ] 记录每个 Worker 的输入、输出、耗时和 token，汇总到一份周报级 Trace。
- [ ] 对比实验：同样的周报任务用单 Agent 一次性完成，比较质量、耗时和成本，验证多 Agent 是否值得。

第一个验收标准：能够只使用本地论文生成一份结构稳定、带真实 arXiv 引用的中文 Markdown 周报；资料不足的主题必须明确说明。多 Agent 不属于第一个小目标。

### 阶段 13：跨会话长期记忆

目标：理解短期对话记忆与长期记忆的区别，以及记忆的写入、召回和过期策略。

- [ ] 设计最小记忆 schema：用户关注的主题、问过的问题摘要、明确表达的偏好，存入 SQLite 新表。
- [ ] 会话结束或每轮结束时，用 LLM 提炼本轮值得长期保存的记忆条目，去重后写入。
- [ ] 新会话开始时召回相关记忆注入系统提示，并在 Trace 中标注注入了哪些记忆。
- [ ] 提供查看和删除记忆的 API 端点，用户可清空自己的记忆。
- [ ] 先手写实现，跑通后对比 Mem0 等现成记忆层，在 `docs/decision-log.md` 记录是否迁移。

验收标准：重启服务后新会话能利用上一会话的记忆改善回答，且用户能查看和删除记忆。

### 阶段 14：MCP 写工具与 Guardrails 护栏

目标：理解输入过滤、输出校验、高危操作审批三层护栏，以及写操作工具的风险控制。

- [ ] 新增 MCP 写工具 `fetch_arxiv_updates(query, max_results)`：手动触发一次 arXiv 增量抓取并同步索引（对应阶段 6 遗留项）。
- [ ] 写工具默认不直接执行：先返回执行计划（将抓取什么、预计新增多少），由调用方二次确认后才真正执行。
- [ ] 为 `/chat` 输入增加基础护栏：长度限制、明显 prompt injection 模式检测，命中时拒绝并记录。
- [ ] 为回答输出增加校验：引用 ID 必须存在于本地库，格式非法时拦截。
- [ ] 护栏触发情况写入执行轨迹，前端可见。

验收标准：写工具未经确认不会改动数据；能演示一次被护栏拦截的注入输入和一次被拦截的非法引用。

### 阶段 15：Langfuse 轨迹级可观测性

目标：理解跨请求 Trace 如何支持线上排障、调用回放、耗时分析和成本观察（承接阶段 8）。

- [ ] 本地部署或注册 Langfuse，把现有执行轨迹映射为 Langfuse 的 trace 和 span。
- [ ] 记录每次检索、重排、LLM 调用的耗时、token 和成本。
- [ ] 支持从一次异常回答回放完整调用链，定位检索、规划、工具或生成阶段的问题。

验收标准：任意一次问答都能在 Langfuse 中回放完整轨迹，并查看主要步骤的耗时和成本。

### 阶段 16：Human-in-the-Loop 人工介入

目标：理解 Agent 在关键节点暂停、呈现状态、等待确认后继续的模式。

- [ ] 选择一个介入点：二次检索前，把改写后的查询呈现给用户，允许确认、修改或跳过。
- [ ] API 设计为可中断会话：`/chat` 返回 `pending_action` 状态，前端渲染确认卡片，用户决定后携带决定继续。
- [ ] 设置超时默认策略：用户不响应时按保守路径继续。
- [ ] 若阶段 11 已完成，用 LangGraph 的 interrupt 机制实现，体会框架对可中断状态的支持。

验收标准：前端能完成一次"暂停 → 用户修改查询 → 继续"的完整问答。

### 阶段 17：Plan-and-Execute 规划执行分离

目标：理解长链任务中"先规划成子任务 DAG、再逐个执行"与 ReAct 逐步探索的区别。

- [ ] 以阶段 12 的周报为载体：先用一次 LLM 调用生成显式计划（主题、每个主题的检索查询、产出结构），计划以 JSON 呈现。
- [ ] 执行前校验计划：主题数量上限、查询合法性，非法则要求重新规划一次。
- [ ] 按计划逐步执行，允许执行中发现证据不足时局部重新规划单个子任务。
- [ ] 对比同一任务下 Plan-and-Execute 与 ReAct 的步数、token 和产出质量。

验收标准：能展示一份显式计划及其执行轨迹，并说明两种模式各自适合的任务类型。

## 10. 第一版完成标准

第一版完成时，应满足：

- arXiv 数据只通过手动批量任务获取。
- 论文元数据已经保存到本地。
- 重复抓取不会产生重复记录。
- 中文问题可以检索英文论文摘要。
- 回答只引用实际检索到的论文。
- 资料不足时会明确拒答。
- React 页面可以展示问答、引用和 Agent Trace。
- 至少有一组可以重复运行的测试问题。
